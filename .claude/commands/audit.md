# /audit Command

Run a codebase security and quality audit using the `code-audit` skill.

## Usage

```
/audit              # Quick scan (5-10 min)
/audit full         # Full audit (30-60 min)
/audit cold         # Cold assessment for new codebase (60-90 min)
/audit types        # Type/schema export manifest for current phase (1-2 min)
```

## Execution

### 1. Read the Skill

```
Read .claude/skills/code-audit/SKILL.md
```

### 2. Determine Mode

| Argument | Mode     | When to Use                                        |
| -------- | -------- | -------------------------------------------------- |
| (none)   | Quick    | After completing a phase                           |
| `full`   | Full     | Before shipping, pre-PR                            |
| `cold`   | Cold     | First look at new/acquired codebase                |
| `types`  | Manifest | At phase end, BEFORE `/init-phase N+1` (mandatory) |

**`types` mode is a fast, narrow scan** — it does not run security or quality checks. It only enumerates exported types/schemas for the next phase to consume. Run it in addition to (not instead of) the standard quick scan at phase boundaries.

### 3. Execute Phases

**Quick Scan:** Phases 1, 2, 3 (partial)
**Full Audit:** Phases 1-6
**Cold Assessment:** Phases 1-6 + architecture mapping
**Types Manifest:** Skip phases — run the Types Mode procedure below.

### 3a. Types Mode Procedure

Run this when invoked as `/audit types`. It produces `specs/phases/PHASE-N-type-manifest.md` for the current phase and exits. The manifest is the contract the next phase reads to avoid reinventing types.

#### Step 1: Determine current phase

```bash
PHASE=$(cat specs/CURRENT_PHASE)
MANIFEST="specs/phases/PHASE-${PHASE}-type-manifest.md"
```

#### Step 2: Enumerate exported types, interfaces, type aliases

Use Grep (not bash) to find every public export in the canonical type locations and the source tree. Run these searches in parallel:

```
Grep pattern: "^export (interface|type|const|enum) "
glob:    "lib/**/*.ts"
output:  content with -n
```

```
Grep pattern: "^export (interface|type|const|enum) "
glob:    "src/**/*.ts"
output:  content with -n
```

```
Grep pattern: "z\.object\(|z\.discriminatedUnion\(|z\.union\("
glob:    "lib/**/*.ts"
output:  content with -n
```

Filter the results:

- **Include**: anything in `lib/**/types.ts`, `lib/**/schemas.ts`, `lib/**/index.ts`, and any file whose name contains `types`, `schema`, `model`, or `contract`.
- **Include**: any `export const fooSchema = z.object(...)` regardless of file location.
- **Exclude**: `*.test.ts`, `*.spec.ts`, `__tests__/`, files in `node_modules/`, `.next/`, `dist/`, `coverage/`.
- **Exclude**: type-only re-exports (`export type { Foo } from "./bar"`) — they point to the canonical definition, not a new one.

#### Step 3: Group exports by module

For each module under `lib/` (e.g. `lib/orchestration`, `lib/users`, `lib/billing`), collect:

- File path
- All `export type ...` declarations
- All `export interface ...` declarations
- All `export const xxxSchema = z.object(...)` declarations
- All `export enum ...` declarations
- A one-line description per export, inferred from the surrounding JSDoc comment if present, or the field shape if not

If a type and its Zod schema share a base name (e.g. `User` and `userSchema`), group them on one line.

#### Step 4: Detect duplicates and overlaps

Cross-reference exports across modules. For each type name that appears more than once, or for each pair where the field set is ≥80% identical, write a `⚠️ Possible duplicate` line in the manifest with both file paths. The next phase agent should treat these as candidates to consolidate, not as two valid types.

#### Step 5: Write the manifest file

Use the Write tool to create `specs/phases/PHASE-${PHASE}-type-manifest.md` with this structure:

````markdown
# Phase {N} Type Manifest

**Generated:** YYYY-MM-DD HH:MM
**Source:** Auto-generated by `/audit types` — do not edit by hand.
**Read by:** `/init-phase {N+1}` and every ticket in subsequent phases.

> **Phase {N+1} agents: Before defining ANY type or Zod schema, search this file for the concept name. If it exists, import it from the file path listed below. Do not redeclare.**

## How to import

```typescript
import type { User } from "@/lib/users/types";
import { userSchema } from "@/lib/users/types";
```
````

For API routes, derive request/response shapes from the canonical schema:

```typescript
import { userSchema } from "@/lib/users/types";
const updateUserInput = userSchema.pick({ name: true, email: true });
```

---

## Module: lib/<module-name>

**Source files:** `lib/<module-name>/types.ts`, `lib/<module-name>/schemas.ts`

| Export             | Kind | Description                                    |
| ------------------ | ---- | ---------------------------------------------- |
| `User`             | type | A registered user with profile and auth fields |
| `userSchema`       | Zod  | Validation schema mirroring `User`             |
| `UserRole`         | enum | `admin` \| `member` \| `guest`                 |
| `CreateUserInput`  | type | Subset of `User` accepted by the create API    |
| `createUserSchema` | Zod  | Validation for `CreateUserInput`               |

[Repeat for every module discovered under lib/]

---

## Possible Duplicates (⚠️ review before next phase)

| Concept      | Locations                                                        | Notes                                         |
| ------------ | ---------------------------------------------------------------- | --------------------------------------------- |
| `Conditions` | `lib/orchestration/types.ts:14`, `src/api/conditions/route.ts:8` | Same shape — consolidate into the lib version |

(If none, write "None detected.")

---

## Coverage

- Modules scanned: N
- Types/interfaces exported: N
- Zod schemas exported: N
- Enums exported: N
- Possible duplicates flagged: N

````

#### Step 6: Append a pointer to the build log

Add a one-line entry to `progress/build-log.md`:

```markdown
### Phase {N} Type Manifest Generated
**Date:** YYYY-MM-DD HH:MM
**File:** specs/phases/PHASE-{N}-type-manifest.md
**Exports:** N types, M schemas, K enums
````

#### Step 7: Report to console

Print a short summary:

```
Type manifest written: specs/phases/PHASE-{N}-type-manifest.md
- Modules: N
- Exports: N types, M schemas
- Duplicates flagged: K
Next: run /init-phase {N+1} — it will consume this manifest automatically.
```

Then exit. Do not run the standard quick scan phases.

### 4. Run the Commands

Execute the bash commands from each phase in the skill file. Collect output.

### 5. Generate Report

**Quick Scan Output:**

```markdown
# Quick Audit: [Project Name]

Date: [timestamp]
Mode: Quick Scan

## Status: [PASS / FAIL]

### Blockers (must fix)

- [list or "None"]

### Warnings (should fix)

- [list or "None"]
```

**Full Audit Output:**

```markdown
# Full Audit Report: [Project Name]

Date: [timestamp]

## Executive Summary

- Overall Health: [1-10]
- Security Posture: [Strong/Adequate/Weak/Critical]
- Code Quality: [High/Medium/Low]
- Recommendation: [Ship/Fix then ship/Major rework needed]

## Critical Issues

[table of issues]

## High Priority Issues

[table of issues]

## Metrics Summary

[metrics table]
```

### 6. Save Report

Save report to `progress/audit-[date].md`

### 7. Decision Gate

| Result                | Action                                   |
| --------------------- | ---------------------------------------- |
| Critical issues found | Block. Do not proceed.                   |
| High issues found     | Flag for human review before proceeding. |
| Medium/Low only       | Log and continue.                        |

## Integration with Phases

This command is automatically suggested:

- After `/init-phase` completes all tickets
- Before creating a PR for human review
- When running `/work` and completing final ticket in a phase

**⚠️ CRITICAL: Run `/audit` after Phase 1 and Phase 2, not just at the end.**

Issues caught early are 10x cheaper to fix. Waiting until Phase 4 to audit allows problems to compound:

- Test gaps become normalized ("we didn't test the other phases either")
- Bad patterns get copied to new code
- Technical debt accumulates
- Refactoring scope grows

| Phase             | Audit Requirement                                                       |
| ----------------- | ----------------------------------------------------------------------- |
| Phase 1 complete  | Run `/audit` then `/audit types` — catches setup issues, missing tests  |
| Phase 2 complete  | Run `/audit` then `/audit types` — catches pattern drift, security gaps |
| Phase 3+ complete | Run `/audit` then `/audit types` — standard end-of-phase check          |
| Before PR         | Run `/audit full` — comprehensive pre-merge check                       |

**`/audit types` is mandatory at every phase boundary** before running `/init-phase N+1`. The next phase's tickets will not see prior exports without it.
